Backend Code Highlights

Slide 10

Idempotent Batch Upserts

In server.js, attendance markings utilize SQLite's native upsert capabilities.

  • Prevents duplicate entries on consecutive teacher submissions.
  • Enforces atomicity by running statements inside database transaction blocks.
  • Automatically resets absent reason fields if a student status transitions from "Absent" to "Present".
// server.js - Batch marking route
await run(
  `INSERT INTO attendance 
   (student_id, date, status, absent_reason, marked_by, timestamp) 
   VALUES (?, ?, ?, ?, ?, ?)
   ON CONFLICT(student_id, date) DO UPDATE SET 
     status = excluded.status, 
     absent_reason = excluded.absent_reason,
     timestamp = excluded.timestamp`,
  [student_id, date, status, reason, marked_by, timestamp]
);

Key Schema Decision: Defined a composite unique key `UNIQUE(student_id, date)` on the attendance table. This maintains data consistency at the SQL layer, preventing duplicate student records on the same calendar date.

Technical Challenges & Solutions

Slide 12

Challenge: Safe Concurrent marking

Multiple teachers marking the same class attendance on identical dates could overwrite files or duplicate rows.

Risk: Database corruption, mismatched attendance percentage metrics, and duplicate rows.

Resolution: Composite Keys & Upserts

Enforced composite indices at the database schema layer and implemented idempotency on the API server.

Outcome: SQLite's native `ON CONFLICT` acts as a guard, turning double-marking requests into safe, non-corruptive updates.

Technical Challenges & Solutions

Slide 13

Challenge: Cost-Free AI Analysis

Integrating remote cloud LLM APIs introduced recurring API billing costs, token limitations, and mandatory web internet connections.

Risk: Unstable response latency, connection dropouts, and hosting charges.

Resolution: Local NLP SQL Engine

Built a localized JavaScript keyword parsing routing engine mapping questions to direct SQLite query filters.

Outcome: Fast, 100% offline analysis of database dropouts and admissions FAQs with zero API costs.

Learnings, Limitations & Future Scope

Slide 15

Key Learnings

Gained hands-on experience in SQLite schema design, SQL transactions, asynchronous Express API development, and print CSS styling layouts.

Current Limitations

SQLite operates on a local file, restricting concurrent writes across servers. The prototype currently lacks role-based user login access checks.

Future Scope

Migrate to PostgreSQL for cloud deployment, add JWT token-based session auth, and hook SMS/WhatsApp gateway APIs for instant parent notifications.
